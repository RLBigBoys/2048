# 2048 RL Project

Этот проект посвящен обучению агента для игры 2048. Ниже описано, как проект развивался по шагам, что было сделано на каждом этапе, и как по ходу работы менялось поведение агента и метрики.

## 1. Первая версия проекта

В начале в проекте была только базовая среда 2048 и ручной GUI для отладки.

Что было реализовано:
- чистая NumPy-среда для 2048
- действия `UP`, `DOWN`, `LEFT`, `RIGHT`
- корректная логика сдвига и merge
- спавн новых тайлов
- определение терминального состояния
- `pygame`-GUI для ручной игры

На этом этапе проекта обучения еще не было. Основная задача была убедиться, что сама игровая механика работает правильно.

## 2. Проверка и исправление среды

Дальше проект был проверен именно как RL-среда.

Что было проверено:
- не меняется ли доска после невалидного хода
- не спавнится ли новый тайл после невалидного хода
- правильно ли считаются valid actions
- правильно ли определяется terminal state
- не ломается ли логика merge на краевых случаях

Итог:
- базовая механика среды оказалась корректной
- проблема с “лишней четверкой” в GUI оказалась не багом merge, а обычным спавном нового тайла после валидного хода

## 3. Первые признаки состояния: блоки 2x2

Следующий шаг был связан с уменьшением пространства состояний.

Изначальная идея:
- не хранить полное состояние всей доски как табличное
- разбивать доску на локальные шаблоны `2x2`
- использовать эти шаблоны как признаки для табличного обучения

Сначала было реализовано 8 блоков `2x2`, но затем был добавлен центральный блок, и разбиение стало полным:
- `3 x 3` перекрывающихся окна `2x2`
- всего `9` блоков `2x2`

Это был первый шаг к признаковому представлению состояния.

## 4. Первый агент и первая схема выбора действия

После этого был добавлен обучаемый агент.

Что было сделано:
- появился `LocalMajorityAgent`
- для каждого локального признака стала храниться табличная `Q`-оценка действий
- были реализованы два режима обучения:
  - `value_iteration`
  - `policy_iteration`

Первая схема глобального выбора действия была такой:
- каждый блок `2x2` голосует за лучший ход
- глобальный ход выбирается по `majority vote`

Почему это было сделано:
- идея была в том, чтобы решение принималось локально в маленьких шаблонах
- а затем собиралось в одно глобальное действие

## 5. Learning curve, сохранение модели и визуализация

После этого проект стал более “рабочим” как полноценный pipeline.

Было добавлено:
- сохранение learning curve в CSV
- сохранение графиков learning curve в PNG
- сохранение обученного агента в `artifacts/agent_policy.pkl`
- отдельная оценка агента без обучения
- отдельный скрипт [`evaluate.py`](/home/romanbranovets/RLCourse/2048/evaluate.py), который загружает сохраненную модель и запускает игру обученного агента

Также в проекте появилась визуализация:
- ручной режим GUI
- визуализация во время обучения
- визуализация во время evaluation

## 6. Переход от majority vote к более сильным агрегаторам

Когда началось реальное обучение, стало видно, что простой `majority vote` слишком слаб.

Что наблюдалось:
- агент мог делать локально разумные ходы
- но не удерживал глобальную стратегию
- learning curve быстро выходила на плато
- `max_tile` обычно упирался в `256` или `512`

После этого были опробованы другие способы выбора действия.

### 6.1. Сумма Q

Была реализована схема:
- не считать просто число голосов
- а суммировать Q-оценки по всем локальным признакам

Идея была в том, чтобы сохранить силу предпочтений, а не только бинарный голос.

### 6.2. Weighted majority vote

Следующим шагом стал `weighted_majority_vote`.

Что он делает:
- каждый признак по-прежнему голосует за одно действие
- но вклад голоса зависит от важности признака

Вес зависит от:
- максимального значения в шаблоне
- суммы значений в шаблоне

Именно этот режим сейчас является основным:
- `action_selection_mode = "weighted_majority_vote"`

## 7. Расширение признакового пространства

Когда стало видно, что только `2x2` шаблонов недостаточно, признаки были расширены.

Сначала:
- `9` блоков `2x2`

Потом были добавлены:
- `4` строки
- `4` столбца

Затем добавлены более длинные локальные шаблоны:
- `6` блоков `2x3`
- `6` блоков `3x2`

Итоговое представление состояния сейчас:
- `9` блоков `2x2`
- `6` блоков `2x3`
- `6` блоков `3x2`
- `4` строки
- `4` столбца

Всего:
- `29` активных признаков на одну доску

Важно:
- признаки теперь tagged по типу, например `("block", tuple)` или `("row", tuple)`
- это нужно, чтобы одинаковые tuples не смешивались между разными типами признаков

## 8. Изменение reward function под реальную цель

По мере обучения стало ясно, что задача проекта не просто “играть неплохо”, а именно дойти до `2048`.

Из-за этого reward function несколько раз усиливалась и менялась.

Сейчас в награде есть:
- штраф за невалидный ход
- reward за merge через `reward_score_scale`
- маленький бонус за изменение числа пустых клеток
- бонус за рост максимального тайла
- большой бонус за достижение `target_tile`
- мягкий штраф за долгую стагнацию без роста max tile
- corner-based bonus/penalty за положение максимального тайла

### 8.1. Явная цель 2048

Были добавлены:
- `reward_target_tile_bonus`
- `terminate_on_target_tile`

Это позволило сделать достижение `2048` не просто побочным эффектом, а явной частью постановки задачи.

### 8.2. Штраф за долгую стагнацию

Было замечено, что агент может долго крутиться без роста `max_tile`.

Поэтому были добавлены:
- `reward_stagnation_penalty`
- `stagnation_penalty_after_steps`

Среда теперь считает число шагов без роста максимального тайла и после порога начинает мягко штрафовать.

### 8.3. Corner strategy

Так как для 2048 важна стратегия удержания большого тайла в одном углу, были добавлены:
- `target_corner`
- `reward_max_tile_in_corner_bonus`
- `reward_max_tile_out_of_corner_penalty`

Теперь среда дополнительно поощряет ситуации, когда максимальный тайл находится в выбранном углу.

## 9. Epsilon decay

Изначально exploration был постоянным:
- `epsilon` оставался одинаковым весь train

Позже стало видно, что это мешает закреплению стратегии.

Поэтому был добавлен `epsilon decay`:
- `train_epsilon` — стартовое значение
- `train_epsilon_end` — конечное значение
- `use_epsilon_decay` — включает/выключает decay

Сейчас `epsilon` линейно уменьшается по эпизодам.

## 10. Что показывали метрики по ходу работы

Ниже кратко описано, как менялось поведение агента по learning curves.

### Начальные эксперименты

В первых версиях:
- reward быстро выходил на плато
- score почти не рос
- `max_tile` в основном застревал на `128-256`

Это показывало, что агент научился только локальным merge, но не длинной стратегии.

### После добавления строк, столбцов и длинных шаблонов

После перехода к признакам:
- `2x2 + rows + cols`
- затем `2x2 + 2x3 + 3x2 + rows + cols`

метрики стали лучше:
- score стал расти заметно устойчивее
- `max_tile` стал чаще доходить до `256`
- затем стал чаще встречаться `512`

Это был первый признак того, что агент действительно начал использовать более глобальный контекст доски.

### После epsilon decay

После добавления `epsilon decay`:
- рост score стал более плавным
- редкие хорошие эпизоды стали появляться чаще
- обучение перестало быть полностью “шумовым”

### После corner strategy и anti-stagnation reward

После добавления:
- corner bonus
- stagnation penalty

стало видно, что:
- score продолжает расти
- `max_tile` до `512` добирается уже заметно чаще
- но до `1024` и `2048` агент пока не дошел стабильно

Проект уже ушел далеко от “простой среды для 2048” и превратился в полноценную RL-постановку с:
- собственной reward function
- сложным набором признаков
- табличным локальным агентом
- визуализацией и оценкой

При этом главный текущий результат такой:
- агент уже научился играть лучше случайного и лучше ранних версий
- агент стабильно улучшает score
- агент начал чаще достигать `512`
- но обучение до `2048` пока еще не достигнуто

То есть проект сейчас находится в промежуточной, но уже рабочей точке:
- инфраструктура готова
- среда и обучение работают
- качество стало лучше
- следующий фокус — пробить потолок `512/1024` и довести поведение до устойчивого достижения `2048`

## 13. Как запускать

Обучение:

```bash
PYTHONPATH=src python main.py --mode train
```

```bash
PYTHONPATH=src python main.py --mode gui
```

```bash
PYTHONPATH=src python evaluate.py
```

Без визуализации:

```bash
PYTHONPATH=src python evaluate.py --no-visualize
```
